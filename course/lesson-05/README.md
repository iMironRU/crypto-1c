# Урок 05. Шифрование: сейф для данных

> ⏱ Время: ~45 минут  
> Что нужно до начала: Уроки 01–04, тестовый сертификат в личном хранилище  
> Что будет после: умеете зашифровать строку и файл в 1С, понимаете как хранить зашифрованное в базе и расшифровывать обратно

## Зачем это знать

Представьте сейф с двумя отверстиями: через первое можно только положить, через второе — только взять. Ключ от второго есть только у хозяина. Это суть шифрования: зашифровать может тот, у кого есть подходящий сертификат получателя; расшифровать — тот, у кого есть доступ к соответствующему закрытому ключу.

Задача: хранить в базе паспортные данные сотрудников. Закон о персональных данных требует защиты. Хеш не подходит: он не предназначен для восстановления исходных данных, и для нормального криптографического алгоритма это практически невыполнимо. Нужно шифрование: зашифровали → расшифровали → исходные данные.

Или другой сценарий: передать контрагенту конфиденциальный документ по обычной почте. Шифруете файл, отправляете. Расшифровать сможет тот, у кого есть соответствующий закрытый ключ.

Хеш (уроки 02–03) — необратимый отпечаток. Шифрование — обратимое преобразование. Это принципиальное различие.

## Как это работает

### Ключи и сертификат

В уроке 04 вы уже работали с сертификатом. Теперь используем его для шифрования.

Сертификат **содержит открытый ключ** — именно он используется для шифрования. Закрытый ключ в сертификате не хранится: он находится в контейнере криптопровайдера, на токене или другом защищённом носителе — там, куда у владельца есть доступ.

**Зашифровать** — открытым ключом (из сертификата).  
**Расшифровать** — закрытым ключом (должен быть доступен тому, кто расшифровывает).

### Гибридное шифрование

Вот рабочая аналогия: вы кладёте документ в сейф с одноразовым кодом, а сам код запечатываете в маленький конверт для получателя. Получатель вскрывает конверт, узнаёт код, открывает сейф.

Асимметричный алгоритм (например, RSA) напрямую подходит только для небольших данных: размер ограничен длиной ключа. Поэтому платформа использует гибридный подход:

1. Генерируется случайный симметричный ключ сессии (одноразовый код)
2. Данные шифруются симметричным ключом (документ — в сейф)
3. Ключ сессии шифруется на открытом ключе получателя (код — в конверт)
4. Всё упаковывается в **CMS-конверт** (стандарт CMS/PKCS#7 EnvelopedData)

При расшифровке платформа распаковывает конверт: если закрытый ключ доступен текущему пользователю, находит зашифрованный ключ сессии, расшифровывает его и расшифровывает данные. Для кода это выглядит как один вызов.

Именно поэтому зашифрованный файл как правило **больше** исходного: CMS-конверт добавляет метаданные, идентификатор сертификата и зашифрованный ключ сессии — несколько сотен дополнительных байт.

### Как 1С выполняет шифрование

```
МенеджерКриптографии.Зашифровать(ДвоичныеДанные, Сертификат)  → ДвоичныеДанные (CMS)
МенеджерКриптографии.Расшифровать(ДвоичныеДанные)             → ДвоичныеДанные
```

`Зашифровать` принимает сертификат (или массив сертификатов — для нескольких получателей).  
`Расшифровать` ищет подходящий закрытый ключ самостоятельно — по метаданным внутри CMS-конверта.

## Пример 1. Зашифровать строку

```bsl
// Требования:
// - Платформа 1С 8.3.6+
// - Windows
// - Тестовый сертификат в личном хранилище (см. docs/certificates.md)
// - Замените значение Отпечаток на ваш из Урока 04, Пример 2

Попытка

    Отпечаток = "A1B2C3D4E5F6..."; // вставьте ваш отпечаток без пробелов

    // Подключаем провайдер и получаем сертификат по отпечатку
    МенеджерКрипто = Новый МенеджерКриптографии(
        "Microsoft Strong Cryptographic Provider", "", 1
    );
    Сертификат = МенеджерКрипто.ПолучитьСертификат(Отпечаток);

    // Шаг 1. Строка → ДвоичныеДанные (кодировка UTF-8)
    ИсходнаяСтрока = "Секретные данные: ИНН 7700000001, паспорт 4500 000001";
    ИсходныеДвоичные = ПолучитьДвоичныеДанныеИзСтроки(ИсходнаяСтрока, "UTF-8");
    Сообщить("Исходный размер (байт): " + ИсходныеДвоичные.Размер());

    // Шаг 2. Зашифровать открытым ключом из сертификата → CMS-конверт
    ЗашифрованныеДвоичные = МенеджерКрипто.Зашифровать(ИсходныеДвоичные, Сертификат);
    Сообщить("Зашифрованный размер (байт): " + ЗашифрованныеДвоичные.Размер());
    // Размер больше исходного: CMS-конверт добавляет метаданные и зашифрованный ключ сессии

    // Шаг 3. Для хранения в базе → переводим в Base64
    ЗашифрованноеBase64 = Base64Строка(ЗашифрованныеДвоичные);
    Сообщить("Для хранения в базе (Base64): " + Лев(ЗашифрованноеBase64, 60) + "...");

    // Скопируйте строку — она понадобится в Примере 2
    Сообщить("=== Полная строка Base64 ===");
    Сообщить(ЗашифрованноеBase64);

Исключение
    Сообщить("Ошибка: " + ОписаниеОшибки());
    // Частые причины:
    // - сертификат не найден (неверный отпечаток)
    // - сертификат не предназначен для шифрования (в параметрах сертификата
    //   должен быть указан признак "Шифрование ключей" / KeyEncipherment)
КонецПопытки;
```

Запустите пример — скопируйте полную строку Base64 из вывода, она понадобится в Примере 2.

> **В боевой системе (ГОСТ):** замените инициализацию провайдера:
> ```bsl
> МенеджерКрипто = Новый МенеджерКриптографии(
>     "Crypto-Pro GOST R 34.10-2012 Cryptographic Service Provider", "", 80
> );
> ```
> Точное имя и тип провайдера зависят от версии КриптоПро — проверьте через список провайдеров из Урока 04, Пример 1. Остальная логика кода та же: получить сертификат, зашифровать двоичные данные и сохранить результат в Base64.

## Пример 2. Расшифровать строку

Продолжение Примера 1: берём зашифрованную строку из его вывода и расшифровываем обратно.

```bsl
// Требования:
// - Платформа 1С 8.3.6+
// - Windows
// - Тот же сертификат с закрытым ключом, что использовался в Примере 1
// - Замените ЗашифрованноеBase64 на строку из вывода Примера 1

Попытка

    // Берём зашифрованные данные из Примера 1 (или из базы)
    ЗашифрованноеBase64 = "MIIB..."; // вставьте полную строку Base64 из Примера 1

    МенеджерКрипто = Новый МенеджерКриптографии(
        "Microsoft Strong Cryptographic Provider", "", 1
    );
    // Для ГОСТ: замените инициализацию провайдера как в Примере 1

    // Шаг 1. Base64 → ДвоичныеДанные
    ЗашифрованныеДвоичные = Base64Значение(ЗашифрованноеBase64);

    // Шаг 2. Расшифровать
    // Платформа читает CMS-конверт и ищет подходящий закрытый ключ среди ключей,
    // доступных текущему пользователю через указанный криптопровайдер
    РасшифрованныеДвоичные = МенеджерКрипто.Расшифровать(ЗашифрованныеДвоичные);

    // Шаг 3. ДвоичныеДанные → строка (та же кодировка, что при шифровании)
    РасшифрованнаяСтрока = ПолучитьСтрокуИзДвоичныхДанных(РасшифрованныеДвоичные, "UTF-8");
    Сообщить("Расшифровано: " + РасшифрованнаяСтрока);

Исключение
    Сообщить("Ошибка: " + ОписаниеОшибки());
    // Частые причины:
    // - закрытый ключ не найден: зашифровали другим сертификатом или на другой машине
    // - нет прав доступа к закрытому ключу
    // - строка Base64 передана не полностью или повреждена
КонецПопытки;
```

При расшифровке не нужно указывать ни отпечаток, ни сертификат — платформа находит нужный закрытый ключ по метаданным внутри CMS-конверта.

> **В боевой системе (ГОСТ):** замените инициализацию провайдера как в Примере 1.

## Пример 3. Зашифровать файл

Для файлов подход тот же — разница только в источнике данных. Создадим файл из строки, зашифруем и расшифруем, чтобы путь через `ДвоичныеДанные` был виден от начала до конца.

```bsl
// Требования:
// - Платформа 1С 8.3.6+
// - Windows
// - Тестовый сертификат (см. docs/certificates.md)
// - Папка C:\Temp должна существовать
// - Замените Отпечаток на ваш

Попытка

    Отпечаток        = "A1B2C3D4E5F6..."; // ваш отпечаток
    ИсходныйФайл     = "C:\Temp\secret.txt";      // исходный файл
    ЗашифрованныйФайл = "C:\Temp\secret.txt.enc"; // зашифрованный результат

    МенеджерКрипто = Новый МенеджерКриптографии(
        "Microsoft Strong Cryptographic Provider", "", 1
    );
    // Для ГОСТ: замените инициализацию провайдера как в Примере 1

    Сертификат = МенеджерКрипто.ПолучитьСертификат(Отпечаток);

    // Подготовка: Строка → ДвоичныеДанные → записываем в файл
    ТекстФайла = "Конфиденциально: данные для передачи контрагенту";
    ТекстФайлаДвоичные = ПолучитьДвоичныеДанныеИзСтроки(ТекстФайла, "UTF-8");
    ТекстФайлаДвоичные.Записать(ИсходныйФайл);

    // Шаг 1. Читаем файл как ДвоичныеДанные — без преобразования в строку
    ИсходныеДвоичные = Новый ДвоичныеДанные(ИсходныйФайл);
    Сообщить("Исходный размер: " + ИсходныеДвоичные.Размер() + " байт");

    // Шаг 2. Зашифровать
    ЗашифрованныеДвоичные = МенеджерКрипто.Зашифровать(ИсходныеДвоичные, Сертификат);
    Сообщить("Зашифрованный размер: " + ЗашифрованныеДвоичные.Размер() + " байт");

    // Шаг 3. Результат → хранение на диске
    ЗашифрованныеДвоичные.Записать(ЗашифрованныйФайл);
    Сообщить("Зашифровано: " + ЗашифрованныйФайл);

    // Вариант хранения в базе: ДвоичныеДанные → Base64
    ЗашифрованноеBase64 = Base64Строка(ЗашифрованныеДвоичные);
    Сообщить("Base64 для хранения в базе: " + Лев(ЗашифрованноеBase64, 60) + "...");

    // --- Расшифровка ---

    // Шаг 4. Читаем зашифрованный файл
    ЗашифрованныеВозврат = Новый ДвоичныеДанные(ЗашифрованныйФайл);

    // Шаг 5. Расшифровать
    РасшифрованныеДвоичные = МенеджерКрипто.Расшифровать(ЗашифрованныеВозврат);

    // Шаг 6. ДвоичныеДанные → строка (та же кодировка, что при записи)
    Содержимое = ПолучитьСтрокуИзДвоичныхДанных(РасшифрованныеДвоичные, "UTF-8");
    Сообщить("Расшифровано: " + Содержимое);

Исключение
    Сообщить("Ошибка: " + ОписаниеОшибки());
КонецПопытки;
```

Полный путь виден явно: `Строка → ДвоичныеДанные → файл → Зашифровать → файл/Base64`. При расшифровке — зеркально в обратную сторону.

> **В боевой системе (ГОСТ):** замените инициализацию провайдера как в Примере 1.

## Истёкший сертификат и расшифровка

`Зашифровать` использует **открытый ключ** из сертификата. `Расшифровать` использует
**закрытый ключ** из контейнера провайдера. Это два разных объекта.

Закрытый ключ не имеет срока действия — он существует в контейнере до явного удаления.
Поэтому данные, зашифрованные до истечения сертификата, расшифровать можно: математически
нужный ключ никуда не делся.

На практике:

- Шифровать **новые** данные на истёкшем сертификате не следует: получатель вправе
  отклонить конверт.
- Расшифровать **ранее зашифрованные** данные — можно.

> ❓ Проверяет ли платформа срок действия сертификата при вызове `Расшифровать` —
> зависит от версии 1С и провайдера. Возможно предупреждение или исключение.
> Проверьте на своей конфигурации.

**Практически:** если под истёкшим сертификатом есть архивные зашифрованные данные —
не удаляйте контейнер с закрытым ключом. Пока есть доступ, экспортируйте его в PFX
(`certmgr.msc` → Личные → Экспорт) и храните в защищённом месте.

Подробно о KeyUsage и жизненном цикле сертификата — в уроке 04.

## Что запомнить

- **Шифрование обратимо** — в отличие от хеширования. Зашифровали → расшифровали = исходные данные.
- **Сертификат содержит открытый ключ** для шифрования. Закрытый ключ хранится отдельно — в контейнере криптопровайдера или на токене — и нужен для расшифровки.
- **CMS-конверт** — стандартный формат результата: данные шифруются симметричным ключом, а сам ключ защищается на открытом ключе получателя. Поэтому результат обычно больше исходных данных.
- **Путь для строки:** `Строка → ДвоичныеДанные → Зашифровать → Base64` для хранения. Обратно: `Base64 → ДвоичныеДанные → Расшифровать → ДвоичныеДанные → Строка`.
- **Платформа сама находит закрытый ключ** при расшифровке — ищет среди ключей, доступных текущему пользователю через указанный криптопровайдер. Явно указывать его не нужно.
