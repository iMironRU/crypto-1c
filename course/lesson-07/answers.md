# Ответы — Урок 07. Криптография в реальных задачах

---

**1. Контроль целостности файла.**

При сохранении вычисляем хеш SHA-256 от файла и сохраняем его в базе
(или передаём вместе с файлом по защищённому каналу). При получении
вычисляем хеш снова и сравниваем с сохранённым.

SHA-256 надёжнее MD5 или SHA-1 — у него нет известных коллизий.
Контрольное значение хранится отдельно от файла (например, в реквизите базы)
или передаётся по другому каналу, иначе подменивший файл подменит и хеш.

---

**2. Хранение токенов в базе.**

Токены нужно **шифровать** — они понадобятся в исходном виде при обращении к API.
Хеш не подходит: он необратим, из хеша токен не восстановить.

Правильная схема: технический сертификат устанавливается на сервере 1С.
Токен шифруется открытым ключом этого сертификата и хранится в базе в виде Base64.
При использовании расшифровывается закрытым ключом — он доступен только на сервере.
Файл базы без закрытого ключа бесполезен для злоумышленника.

---

**3. Подписать и зашифровать для контрагента.**

Порядок: сначала подпись, потом шифрование.

1. Подписываем документ своим закрытым ключом → получаем документ + подпись.
2. Упаковываем оба объекта вместе.
3. Шифруем пакет открытым ключом получателя.

Получатель делает в обратном порядке:
1. Расшифровывает своим закрытым ключом.
2. Проверяет подпись открытым ключом отправителя.

Почему именно так: если сначала зашифровать, а потом подписать —
подпись будет на зашифрованных данных. Кто угодно может переподписать
чужой зашифрованный пакет и выдать его за свой. Подпись на открытых
данных гарантирует что именно эти данные создал конкретный отправитель.

---

**4. HMAC.**

HMAC (Hash-based Message Authentication Code) — хеш, при вычислении которого
используется секретный ключ. Только тот, кто знает ключ, может вычислить
правильный HMAC.

Обычный SHA-256: любой может посчитать хеш от любых данных.
HMAC-SHA256: результат зависит от секретного ключа — без ключа не подделать.

Используется для аутентификации API-запросов: сервер и клиент знают один
секретный ключ. Клиент подписывает запрос HMAC, сервер проверяет.
Перехваченный запрос без знания ключа переподписать невозможно.

---

**5. Проверка действительности сертификата.**

Два условия:
1. Текущая дата позже `ДатаНачала` сертификата.
2. Текущая дата раньше `ДатаОкончания` сертификата.

Дополнительно (не обязательно в учебных примерах, но важно в боевой системе):
проверка цепочки доверия (сертификат подписан доверенным УЦ) и статус отзыва (CRL/OCSP).
