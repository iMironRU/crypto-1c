# Ответы — Урок 06. Электронная подпись: цифровая печать

---

**1. Почему подписывают хеш.**

Асимметричное шифрование медленное — обрабатывать им большой файл целиком
было бы неприемлемо долго. Хеш короткий (32–64 байта), вычисляется быстро
и однозначно идентифицирует содержимое документа.

Если изменить хоть один байт документа — хеш изменится, и при проверке
вычисленный хеш не совпадёт с хешем из подписи. Метод `ПроверитьПодпись()`
выбросит исключение. Именно это гарантирует целостность: нельзя изменить
документ так, чтобы старая подпись осталась действительной.

---

**2. Ключи ЭП vs ключи шифрования.**

| | Электронная подпись | Шифрование |
|-|--------------------|---------  |
| Создание/шифрование | Закрытый ключ подписанта | Открытый ключ получателя |
| Проверка/расшифровка | Открытый ключ подписанта | Закрытый ключ получателя |

В шифровании открытым ключом шифруют (любой может), закрытым — расшифровывают (только владелец).
В ЭП всё наоборот: закрытым ключом подписывают (только владелец), открытым — проверяют (любой).

Это обеспечивает свойство неотрекаемости: подписать мог только владелец закрытого ключа,
а проверить может любой, у кого есть его сертификат.

---

**3. Три способа хранить подпись.**

**Отсоединённая** — подпись в отдельном файле `.sig`/`.p7s`, документ не меняется.
Читается любым просмотрщиком. Для проверки нужны оба файла.
Используется при обмене с госсистемами (ФНС, ФСС). В 1С `Подписать()` создаёт именно её.

**Присоединённая CMS** — документ целиком упаковывается в CMS-контейнер `.p7m`.
Файл больше не является PDF или DOCX — его не откроет Adobe Reader или Word.
Для чтения нужен специальный инструмент (КриптоАРМ и др.). Неудобна для конечного
пользователя именно по этой причине. Применяется редко.

**Встроенная в формат** — подпись встраивается в структуру самого файла.
PDF остаётся валидным PDF: открывается в любом просмотрщике, Adobe Reader
показывает панель со статусом подписи. DOCX остаётся DOCX.
В 1С реализуется через `ЗаписьPDF.ЗаписатьПодпись()` (платформа 8.3.19+).
Наиболее удобный для пользователя вариант.

---

**4. ПЭП, НЭП, КЭП.**

**ПЭП** (простая) — логин/пароль, SMS-код, галочка в форме. Криптографии нет.
Юридическая сила — только по соглашению сторон.

**НЭП** (неквалифицированная) — криптографическая подпись, сертификат от любого УЦ
(в том числе самоподписанный). Юридическая сила — по соглашению сторон.
Примеры из курса (Microsoft Strong CSP, RSA) — это НЭП.

**КЭП** (квалифицированная) — криптографическая подпись на алгоритмах ГОСТ,
сертификат от аккредитованного Минцифры удостоверяющего центра, криптосредство
с сертификатом ФСБ (например, КриптоПро CSP). По закону равнозначна
собственноручной подписи без дополнительных соглашений.

Для работы с ФНС, ФСС, Госуслугами, ЭДО нужна **КЭП**.

---

**5. Нужно ли указывать сертификат при проверке.**

Нет. Сертификат подписанта встроен в CMS-пакет подписи — платформа
извлекает его оттуда самостоятельно и использует для проверки.

Указывать сертификат вручную нужно только если вы хотите дополнительно
проверить, что подписал именно конкретный человек (а не просто что подпись
математически верна). В этом случае после успешной проверки нужно
сравнить отпечаток сертификата из подписи с ожидаемым значением.
