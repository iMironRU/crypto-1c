# Двоичные данные в криптографии 1С

Это не дополнительный материал — это основа. Без понимания этой страницы
криптографический код будет выглядеть как магия.

---

## Почему криптография работает с байтами

Криптографические алгоритмы — это математика над числами.
Любые данные (текст, файл, число) перед обработкой превращаются
в последовательность байт. Результат — тоже байты.

Платформа 1С отражает это через тип `ДвоичныеДанные`.

**Полный путь любой крипто-операции:**

```
Исходные данные          Крипто-операция        Результат
(строка, файл, число)                           (байты)
        │                                           │
        ▼                                           ▼
  ДвоичныеДанные   ──►  [хеш / шифр / подпись]  ──►  ДвоичныеДанные
        │                                           │
        ▼                                           ▼
   (конвертация)                            (конвертация для хранения)
```

Промежуточные шаги нельзя пропустить — платформа не конвертирует
автоматически. Это защита от ошибок, а не неудобство.

## Шаг 1. Строка → ДвоичныеДанные

### Современный способ (платформа 8.3.6+)

```bsl
// ПолучитьДвоичныеДанныеИзСтроки — самый простой способ
// В примерах курса используем UTF-8: одна кодировка = одни байты = одинаковый хеш
Строка = "Привет, мир!";
ДвоичныеДанные = ПолучитьДвоичныеДанныеИзСтроки(Строка, "UTF-8");

// Обратно:
СтрокаОбратно = ПолучитьСтрокуИзДвоичныхДанных(ДвоичныеДанные, "UTF-8");
```

### Через временный файл (универсальный, все версии)

```bsl
// Используется когда нужно точно контролировать кодировку и BOM
Строка = "Привет, мир!";

ИмяФайла = ПолучитьИмяВременногоФайла();
// ИспользоватьBOM — четвёртый параметр, не пятый. Ложь = без BOM.
ЗаписьТекста = Новый ЗаписьТекста(ИмяФайла, "UTF-8", , Ложь);
ЗаписьТекста.Записать(Строка);
ЗаписьТекста.Закрыть();

ДвоичныеДанные = Новый ДвоичныеДанные(ИмяФайла);
УдалитьФайлы(ИмяФайла);
```

> ⚠️ **Кодировка имеет значение.** Одна и та же строка в UTF-8 и в
> Windows-1251 — разные байты, и хеш будет разным. Всегда фиксируйте
> кодировку явно. В примерах курса используем **UTF-8 без BOM** —
> если внешний протокол не требует другого.

## Шаг 2. Что делать с результатом (ДвоичныеДанные на выходе)

Результат любой крипто-операции — `ДвоичныеДанные`. Их нельзя
напрямую сохранить в строковый реквизит или показать пользователю.
Нужно представление: **Base64** или **HEX**.

### Base64 — компактнее HEX, для хранения и передачи

```bsl
ДвоичныеДанные = ...; // результат хеша, подписи, шифрования

// Двоичные → строка Base64
СтрокаBase64 = Base64Строка(ДвоичныеДанные);
// Пример: "47DEQpj8HBSa+/TImW+5JCeuQeRkm5NMpJWZG3hSuFU="

// Строка Base64 → двоичные обратно
ДвоичныеДанные = Base64Значение(СтрокаBase64);
```

**Когда использовать Base64:**
- Хранение хешей и подписей в строковых реквизитах базы данных
- Передача двоичных данных в JSON/XML
- Собственные HTTP-заголовки с двоичным содержимым
- Хранение зашифрованных данных как текст

### HEX — для отображения, логов, сравнения

```bsl
// ДвоичныеДанные → HEX-строка через буфер
Функция ДвоичныеВHex(ДвоичныеДанные)
    Буфер = Новый БуферДвоичныхДанных(ДвоичныеДанные.Размер());
    ДвоичныеДанные.КопироватьВБуфер(Буфер, 0);

    Результат = "";
    Для й = 0 По Буфер.Размер() - 1 Цикл
        // Каждый байт → два hex-символа, с ведущим нулём
        Результат = Результат + Прав("0" + НРег(Формат(Буфер.ПолучитьБайт(й), "ЧС=16")), 2);
    КонецЦикла;

    Возврат Результат;
КонецФункции

// Пример результата: "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934c"
```

**Когда использовать HEX:**
- Вывод хеша пользователю
- Логирование для отладки
- Сравнение результатов с внешними системами (openssl, certmgr)
- Отпечаток сертификата

## Шаг 3. Как хранить двоичные данные в базе 1С

### Вариант А: Строковый реквизит + Base64 (рекомендуется для хешей)

Подходит для: хеши паролей, отпечатки сертификатов, небольшие подписи.

```bsl
// При сохранении
Объект.ХешПароля = Base64Строка(ДвоичныеДанные); // строка длиной ~44 символа для SHA256

// При чтении
ДвоичныеДанные = Base64Значение(Объект.ХешПароля);
```

Реквизит в метаданных: `Тип = Строка`, длина 100–500 символов
(SHA256 в Base64 = 44 символа, SHA512 = 88, подпись ГОСТ = ~128).

### Вариант Б: ХранилищеЗначений (для больших двоичных данных)

Подходит для: зашифрованные файлы, большие подписи, сертификаты.

```bsl
// При сохранении
Объект.ЗашифрованныеДанные = Новый ХранилищеЗначений(ДвоичныеДанные, Новый СжатиеДанных(0));

// При чтении
ДвоичныеДанные = Объект.ЗашифрованныеДанные.Получить();
```

Реквизит в метаданных: `Тип = ХранилищеЗначений`.

> ⚠️ **ХранилищеЗначений — не для маленьких данных.** Оно создаёт
> overhead в базе. Хеш пароля (44 символа) храните как строку,
> а не как ХранилищеЗначений.

### Вариант В: Отдельная таблица подписей (для документооборота)

Когда подписей много, у одного документа несколько подписантов,
нужна история — выносите в отдельный регистр сведений:

```
Регистр сведений: ПодписиДокументов
├── Измерение: ДокументСсылка (любой тип документа)
├── Измерение: НомерПодписи  (число)
├── Реквизит: Подписант      (СправочникСсылка.Пользователи)
├── Реквизит: ДатаПодписания (Дата)
├── Реквизит: ОтпечатокСертификата (Строка 100)
├── Реквизит: Подпись        (ХранилищеЗначений)  ← сами байты подписи
└── Реквизит: СертификатBase64 (Строка неогр.)   ← публичный ключ
```

## Шпаргалка: что куда

| Что | Представление | Где хранить | Длина реквизита |
|-----|--------------|-------------|-----------------|
| Хеш SHA256 | Base64 | Строка | 44 символа |
| Хеш SHA512 | Base64 | Строка | 88 символов |
| Соль пароля | Base64 или HEX | Строка | 50–100 символов |
| Отпечаток сертификата | HEX | Строка | 40–64 символа |
| Подпись ГОСТ | Base64 | Строка неогр. или ХранилищеЗначений | ~200–500 |
| Зашифрованный файл | — (двоичные) | ХранилищеЗначений | — |
| Публичный ключ / сертификат | Base64 | Строка неогр. | — |

## Частые ошибки

### Ошибка 1: сравнение строк вместо двоичных

```bsl
// ❌ Неправильно — строка может содержать лишние пробелы или переводы строк
Если ХешBase64 = ХранимыйХеш Тогда ...

// ⚠️ Опасно для Base64 — НРег меняет значение ('A' и 'a' — разные символы алфавита Base64)
// НРег допустим только для HEX (символы 0-9 и a-f не зависят от регистра)
Если НРег(СокрЛП(ХешBase64)) = НРег(СокрЛП(ХранимыйХеш)) Тогда ...

// ✅ Правильно для Base64 — убираем пробелы, регистр не трогаем
Если СокрЛП(ХешBase64) = СокрЛП(ХранимыйХеш) Тогда ...

// ✅ Ещё правильнее — сравниваем сами двоичные данные
Если ДвоичныеДанные1 = ДвоичныеДанные2 Тогда ...
```

### Ошибка 2: разная кодировка при хешировании

```bsl
// При сохранении использовали UTF-8
ДД1 = ПолучитьДвоичныеДанныеИзСтроки(Пароль, "UTF-8");

// При проверке случайно написали Windows-1251
ДД2 = ПолучитьДвоичныеДанныеИзСтроки(Пароль, "Windows-1251");

// Хеши не совпадут — пользователь не может войти
// Всегда фиксируйте кодировку константой в общем модуле
```

### Ошибка 3: BOM в начале данных

```bsl
// ❌ С BOM — 3 лишних байта портят хеш
ЗаписьТекста = Новый ЗаписьТекста(ИмяФайла, "UTF-8"); // BOM добавляется по умолчанию

// ✅ Без BOM — ИспользоватьBOM это четвёртый параметр
ЗаписьТекста = Новый ЗаписьТекста(ИмяФайла, "UTF-8", , Ложь);
```
